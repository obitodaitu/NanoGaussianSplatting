================================================================================
HIERARCHICAL LOD + RENDER BUDGET IMPLEMENTATION PLAN
================================================================================

Goal: Constant frame time regardless of total splat count by combining
SparkJS-style top-down LOD traversal with a fixed render budget.

Target: Reduce GaussianSplatRendering_Global from 11.7ms to ~3-4ms.

================================================================================
PROBLEM ANALYSIS (from Unreal Insights profiling)
================================================================================

Current pipeline for 70 proxies (11.7ms total):
  DrawGlobalIndirect:              5.2ms  (44%) - TOO MANY SPLATS RENDERED
  CalcViewDataCompactedGlobal:     1.7ms  (15%) - 75 dispatches
  RadixSortGlobalIndirect:         1.5ms  (13%)
  CompactSplats:                   1.2ms  (10%) - 70 dispatches, iterates ALL splats
  ClusterCulling:                  781us  ( 7%) - 70 dispatches, iterates ALL leaf clusters
  GatherVisibleCount:              553us  ( 5%) - 70 dispatches
  PrepareIndirectArgs:             330us  ( 3%) - 70 dispatches
  CalcDistancesGlobalIndirect:     315us  ( 3%)

ROOT CAUSES:
1. Bottom-up traversal: ClusterCulling launches 1 thread per LEAF cluster,
   walks up parent chain. With 70 proxies x ~5600 leaf clusters = ~392K threads,
   ALL doing serial parent chain walks. Even if a LOD3 parent suffices, every
   leaf descendant still gets a thread.

2. CompactSplats iterates ALL splats (original + LOD) per proxy to check
   visibility bits. With 70 proxies, this means 70 x 719K = 50M threads.

3. No global budget enforcement: each leaf cluster independently decides LOD.
   The total visible splat count is unbounded. With many proxies close to camera,
   the draw call can process millions of splats.

4. Per-proxy dispatch overhead: 70 dispatches x 5 passes = 350 small GPU
   dispatches with significant kernel launch overhead.

================================================================================
SOLUTION: CPU TOP-DOWN TRAVERSAL + RENDER BUDGET
================================================================================

Replace the GPU bottom-up ClusterCulling + CompactSplats with a CPU-side
priority queue traversal (like SparkJS). The CPU already has:
  - Full cluster hierarchy (FGaussianClusterHierarchy in each asset)
  - Camera position and matrices (from FSceneView)
  - Global view of all proxies (can enforce cross-proxy budget)

WHY CPU, NOT GPU:
- Priority queue is trivial on CPU, extremely hard on GPU
- Tree is small: ~6400 nodes per proxy, ~340 nodes traversed per proxy
- 70 proxies x 340 nodes = ~24K priority queue operations = <0.1ms on CPU
- Eliminates 350 GPU dispatches (ClusterCulling + CompactSplats + Gather + PrepareArgs)
- GPU does what it's good at: CalcViewData + Sort + Draw on a pre-selected flat list

================================================================================
PHASE 1: CPU-SIDE HLOD TRAVERSAL (the main change)
================================================================================

--- 1.1 New Class: FHLODTraversal ---

File: Source/GaussianSplatting/Private/HLODTraversal.h/.cpp

Purpose: Per-frame CPU traversal of all proxy hierarchies with a shared
render budget. Outputs a flat "render list" for the GPU.

struct FHLODRenderItem
{
    uint32 ProxyIndex;          // Which proxy this cluster belongs to
    uint32 ClusterIndex;        // Index into that proxy's ClusterBuffer
    uint32 SplatStartIndex;     // Start in PackedSplatBuffer (leaf: original, parent: LOD)
    uint32 SplatCount;          // How many splats to render from this cluster
    bool   bIsLOD;              // true = LOD splats, false = original leaf splats
};

struct FHLODTraversalResult
{
    TArray<FHLODRenderItem> RenderList;  // Flat list of what to render
    uint32 TotalSplatCount;              // Sum of all SplatCounts
    // Per-proxy stats for debug
    TArray<uint32> PerProxySplatCounts;
};

class FHLODTraversal
{
public:
    // Main entry: traverse all proxies, output render list within budget
    static FHLODTraversalResult Traverse(
        const TArray<FGaussianSplatSceneProxy*>& Proxies,
        const FSceneView& View,
        uint32 RenderBudget);  // e.g. 3,000,000

private:
    struct FTraversalNode
    {
        float PixelScale;           // Screen-space size (priority key)
        uint32 ProxyIndex;
        uint32 ClusterIndex;
        uint32 DescendantSplatCount; // Total splats if fully expanded
        uint32 LODSplatCount;        // Splats if rendered at this level
    };
    // Max-heap by PixelScale
};

Algorithm (SparkJS-style priority queue):

    frontier = MaxHeap(by PixelScale)
    for each proxy:
        push root cluster(s) with computed PixelScale

    renderList = []
    totalSplats = 0

    while frontier not empty:
        node = frontier.pop()  // largest screen contribution

        // Termination 1: too small to matter (< 1 pixel)
        if node.PixelScale < minPixelScale:
            // Add node as-is (LOD representation)
            renderList.append(node as LOD)
            totalSplats += node.LODSplatCount
            break  // remaining frontier nodes are even smaller

        // Can we afford to expand?
        childSplatCount = sum of children's LODSplatCounts
        costDelta = childSplatCount - node.LODSplatCount

        if totalSplats + costDelta <= budget:
            // Expand: replace this node with its children
            for each child of node:
                compute child.PixelScale from camera distance + bounding sphere
                if child is leaf:
                    renderList.append(child as original splats)
                    totalSplats += child.SplatCount
                else:
                    frontier.push(child)
        else:
            // Budget exceeded: render this node at current LOD level
            renderList.append(node as LOD)
            totalSplats += node.LODSplatCount

    // Drain remaining frontier as LOD
    while frontier not empty:
        node = frontier.pop()
        renderList.append(node as LOD)
        totalSplats += node.LODSplatCount

    return renderList

PixelScale computation:
    worldRadius = cluster.BoundingSphereRadius * maxScale(LocalToWorld)
    distToCamera = distance(worldCenter, cameraPos)
    pixelScale = (worldRadius / distToCamera) * projectionScale
    // projectionScale = max(Proj[0][0], Proj[1][1]) * 0.5 * viewportHeight

Frustum culling integration:
    Before pushing a node to the frontier, test its bounding sphere against
    the view frustum. Culled nodes are simply not pushed (zero cost).

--- 1.2 New GPU Buffer: RenderListBuffer ---

The CPU traversal outputs FHLODRenderItem array. Upload to GPU as a
StructuredBuffer. Each item tells the GPU exactly which splat range to process.

struct FGPURenderListItem  // 16 bytes, shader-friendly
{
    uint PackedSplatStart;    // Absolute index into PackedSplatBuffer
    uint SplatCount;
    uint ProxyIndex;          // For LocalToWorld lookup
    uint Flags;               // LOD flag, debug info
};

This replaces: ClusterVisibilityBitmap, SelectedClusterBuffer,
LODClusterSelectedBitmap, LODClusterBuffer, VisibleClusterBuffer,
SplatClusterIndexBuffer.

--- 1.3 New Shader: ExpandRenderList.usf ---

Replaces CompactSplats. Instead of iterating ALL splats and checking bitmaps,
this shader iterates only the render list items and writes consecutive splat
indices.

Input:  StructuredBuffer<FGPURenderListItem> RenderListBuffer
        uint RenderListCount
Output: RWStructuredBuffer<uint> CompactedSplatIndices (same as before)

Algorithm:
    // One threadgroup per RenderListItem
    // Threads within group write consecutive indices
    uint itemIndex = GroupID.x;
    if (itemIndex >= RenderListCount) return;

    FGPURenderListItem item = RenderListBuffer[itemIndex];
    uint baseOffset = item.PrefixSumOffset;  // Pre-computed on CPU

    for (uint i = GroupThreadID.x; i < item.SplatCount; i += THREADGROUP_SIZE)
        CompactedSplatIndices[baseOffset + i] = item.PackedSplatStart + i;

The PrefixSumOffset is computed on CPU (trivial exclusive scan of SplatCounts).
No atomics needed. Fully deterministic output order.

--- 1.4 Modified Pipeline Flow ---

BEFORE (per proxy, 70x):
    ClusterCulling -> CompactSplats -> GatherVisibleCount -> PrepareIndirectArgs

AFTER (once globally):
    [CPU] HLODTraversal::Traverse() -> RenderList + TotalSplatCount
    [GPU] ExpandRenderList (1 dispatch, covers all proxies)
    [GPU] CalcViewDataFromRenderList (1 dispatch, covers all proxies)
    [GPU] CalcDistancesGlobal (1 dispatch, unchanged)
    [GPU] RadixSortGlobal (16 dispatches, unchanged)
    [GPU] DrawSplatsGlobal (1 draw call, unchanged)

Dispatch count: 350 dispatches -> 20 dispatches
Expected savings: ~2.9ms from eliminating ClusterCulling + CompactSplats +
    GatherVisibleCount + PrepareIndirectArgs (total was 2.87ms)

--- 1.5 Modified CalcViewData ---

Currently CalcViewDataCompactedGlobal dispatches per proxy (75 dispatches)
because each proxy has its own LocalToWorld matrix and SH data.

New approach: upload per-proxy transforms as a buffer.

Buffer: StructuredBuffer<FProxyTransforms> ProxyTransformBuffer
    struct FProxyTransforms {
        float4x4 LocalToWorld;
        float4x4 WorldToPLY;
        float3 ProxyCameraPosition;  // Camera in PLY space for SH
        // ... other per-proxy params
    };

CalcViewData reads ProxyIndex from a parallel ProxyIndexBuffer (written by
ExpandRenderList alongside CompactedSplatIndices), then loads the correct
transform. Single dispatch for all proxies.

Dispatch count: 75 -> 1
Expected savings: ~1.2ms (dispatch overhead reduction; compute work unchanged)

================================================================================
PHASE 2: RENDER BUDGET ENFORCEMENT (integrated with Phase 1)
================================================================================

--- 2.1 Budget CVar ---

Already have gs.MaxRenderBudget (default 3M). This becomes the hard cap
passed to HLODTraversal::Traverse().

--- 2.2 Quality Distribution ---

The priority queue naturally distributes quality optimally:
- Proxies closer to camera get more budget (larger PixelScale)
- Distant proxies collapse to coarse LOD (fewer splats)
- No per-proxy budget allocation needed; it emerges from the global queue

--- 2.3 Adaptive Budget (future enhancement) ---

Optional: adjust budget per frame to hit a target frame time.
    if (lastFrameGPUTime > targetTime)
        budget *= 0.95
    else
        budget *= 1.02
    budget = clamp(budget, minBudget, maxBudget)

Not needed for initial implementation.

The biggest win (3.7ms) comes from the render budget capping splat count.
Currently Draw processes potentially millions of splats; with a 3M budget
spread across 70 proxies at varying distances, many proxies collapse to
coarse LOD, and the actual rendered count drops to ~1-1.5M.

Sort (1ms savings) and CalcViewData (1.3ms) benefit similarly from reduced
splat count + eliminated per-proxy dispatch overhead.

================================================================================
RISKS & MITIGATIONS
================================================================================

1. CPU TRAVERSAL LATENCY
   Risk: CPU work delays GPU submission
   Mitigation: Traversal is O(N_rendered * logN) ~ 24K ops, measured <0.1ms.
   Can also run async on task thread and present 1-frame-old results.

2. LOD POP-IN
   Risk: Coarse LOD visible when budget is tight
   Mitigation:
   - Exponential blending between LOD levels (fade alpha over frames)
   - User-adjustable budget via CVar

3. MULTI-VIEW (editor viewports, VR)
   Risk: CPU traversal runs per view
   Mitigation: For editor, only traverse for active viewport. For VR, share
   traversal between both eyes (same frustum approximately).

4. ASSET COMPATIBILITY
   Old v5 assets must be re-imported. No backward compatibility maintained
   (source control handles rollback if needed).

5. DYNAMIC PROXY CHANGES
   Risk: Proxies added/removed mid-frame
   Mitigation: Snapshot proxy list at traversal time. New proxies appear next frame.
