================================================================================
  Shared Render Data Refactor Plan
  Goal: Eliminate per-proxy GPU buffer duplication for the same asset
================================================================================

ISSUE
-----
When alt+drag duplicating gaussian splat actors (or spawning multiple actors
referencing the same asset), each FGaussianSplatSceneProxy creates its own
FGaussianSplatGPUResources, which:

  1. Re-reads bulk data from UGaussianSplatAsset (CPU memcpy)
  2. Re-packs 211K+ splats into 16-byte format (CPU compute loop)
  3. Re-loads SH data, cluster data, splat-cluster indices (CPU memcpy)
  4. Creates 15+ GPU buffers with individual RHI allocations (GPU/driver stall)
  5. Uploads all data to GPU (DMA)

Profiling (Unreal Insights) shows:
  - 1,827 CreateRHIBuffer calls
  - 609 AllocatePoolResource calls (192ms each)
  - FlushRenderingCommands blocks game thread for 1.9s total
  - Alt+drag 20 copies freezes the editor for several seconds

Static meshes don't have this problem because UStaticMesh stores
FStaticMeshRenderData on the asset itself, shared across all proxies.


SOLUTION
--------
Create FGaussianSplatRenderData, a shared GPU resource object that lives on
UGaussianSplatAsset. All proxies referencing the same asset share the same
static GPU buffers. Each proxy only creates its own per-instance dynamic
buffers (sort, visibility, compaction).

Classification of buffers:

  SHARED (immutable, loaded once from asset):
    - PackedSplatBuffer + SRV          (16 B/splat, packed pos+rot+scale+color)
    - SHBuffer + SRV                   (spherical harmonics coefficients)
    - ChunkBuffer + SRV                (chunk quantization info)
    - IndexBuffer                      (quad rendering indices)
    - ClusterBuffer + SRV              (cluster hierarchy, Nanite)
    - SplatClusterIndexBuffer + SRV    (splat -> cluster mapping)
    - LODSplatClusterIndexBuffer + SRV (LOD splat -> cluster mapping)
    - ColorTexture + SRV               (already on UGaussianSplatAsset as UTexture2D)

  PER-INSTANCE (camera/transform dependent, recomputed per frame):
    - ViewDataBuffer + UAV + SRV
    - SortDistanceBuffer/Alt + UAV + SRV
    - SortKeysBuffer/Alt + UAV + SRV
    - RadixHistogramBuffer + UAV
    - RadixDigitOffsetBuffer + UAV
    - SortIndirectArgsBuffer + UAV
    - SortParamsBuffer + UAV + SRV
    - ClusterVisibilityBitmap + UAV + SRV
    - SelectedClusterBuffer + UAV + SRV
    - VisibleClusterBuffer/Count + UAV + SRV
    - LODClusterBuffer/Count + UAV + SRV
    - LODClusterSelectedBitmap + UAV + SRV
    - LODSplatTotalBuffer + UAV + SRV
    - LODSplatOutputCountBuffer + UAV + SRV
    - CompactedSplatIndicesBuffer + UAV + SRV
    - VisibleSplatCountBuffer + UAV + SRV
    - IndirectDispatchArgsBuffer + UAV
    - IndirectDrawArgsBuffer + UAV
    - DummyWhiteTexture + SRV (per-proxy fallback)


IMPLEMENTATION STAGES
=====================

Each stage is designed to be independently compilable and testable.
At each stage we add verification logs to confirm correctness before
moving to the next stage.


--------------------------------------------------------------------------------
STAGE 1: Create FGaussianSplatRenderData with CPU-side data caching
--------------------------------------------------------------------------------

Purpose:
  Move the CPU-side data loading and packing from FGaussianSplatGPUResources
  into a new FGaussianSplatRenderData class that lives on UGaussianSplatAsset.
  This eliminates repeated bulk-data reads and packing for the same asset.
  GPU buffer creation still happens per-proxy (unchanged behavior).

Files to create:
  - GaussianSplatRenderData.h
  - GaussianSplatRenderData.cpp

Files to modify:
  - GaussianSplatAsset.h         (add TSharedPtr<FGaussianSplatRenderData>)
  - GaussianSplatAsset.cpp       (add GetOrCreateRenderData())
  - GaussianSplatSceneProxy.h    (remove cached data arrays from GPUResources)
  - GaussianSplatSceneProxy.cpp  (GPUResources reads from RenderData)

What FGaussianSplatRenderData holds (Stage 1):
  - TArray<uint8> PackedSplatData     (packed 16B/splat data)
  - TArray<uint8> SHData              (spherical harmonics)
  - TArray<FGaussianChunkInfo> ChunkData
  - TArray<FGaussianGPUCluster> ClusterData
  - TArray<uint32> SplatClusterIndices
  - TArray<uint32> LODSplatClusterIndices
  - int32 SplatCount, SHBands, ClusterCount, LeafClusterCount
  - int32 LODSplatCount, TotalSplatCount
  - bool bEnableNanite, bHasClusterData, bHasLODSplats
  - EGaussianPositionFormat PositionFormat
  - FCriticalSection InitLock    (thread-safe lazy init)
  - bool bIsInitialized

What FGaussianSplatRenderData does (Stage 1):
  - Initialize(UGaussianSplatAsset*):
      Reads bulk data from asset, packs splats, loads SH/cluster data.
      This is the same work currently done in FGaussianSplatGPUResources::Initialize(),
      but only runs ONCE per asset.

What changes in FGaussianSplatGPUResources (Stage 1):
  - Initialize() no longer reads from asset directly
  - Instead calls Asset->GetOrCreateRenderData() to get shared CPU data
  - Copies the shared cached arrays into its own members for GPU upload
  - InitRHI() still creates all GPU buffers per-proxy (no change)

What changes in UGaussianSplatAsset (Stage 1):
  - New member: TSharedPtr<FGaussianSplatRenderData> RenderData
  - New method: TSharedPtr<FGaussianSplatRenderData> GetOrCreateRenderData()
      Lazy-creates RenderData on first call, returns existing on subsequent calls.
  - Invalidation: When asset data changes (OnAssetChanged delegate, Nanite rebuild),
      RenderData.Reset() to force re-creation.

Verification (Stage 1):
  In FGaussianSplatRenderData::Initialize():
    UE_LOG(LogTemp, Log, TEXT("GaussianSplatRenderData: Created shared CPU data for asset '%s' (%d splats)"),
        *AssetName, SplatCount);

  In UGaussianSplatAsset::GetOrCreateRenderData(), when reusing:
    UE_LOG(LogTemp, Log, TEXT("GaussianSplatRenderData: Reusing existing shared data for asset '%s'"),
        *GetName());

  Expected log when alt+drag duplicating 1 asset 3 times:
    GaussianSplatRenderData: Created shared CPU data for asset 'Llama_V2' (211837 splats)
    GaussianSplatRenderData: Reusing existing shared data for asset 'Llama_V2'
    GaussianSplatRenderData: Reusing existing shared data for asset 'Llama_V2'

  If you see "Created" more than once for the same asset, something is wrong.
  The rendering should look identical to before this change.


--------------------------------------------------------------------------------
STAGE 2: Move static GPU buffers to FGaussianSplatRenderData
--------------------------------------------------------------------------------

Purpose:
  Move the immutable GPU buffer creation from FGaussianSplatGPUResources into
  FGaussianSplatRenderData. This eliminates the main bottleneck: repeated RHI
  buffer allocations and data uploads for the same asset.

What changes in FGaussianSplatRenderData (Stage 2):
  - Becomes an FRenderResource (so it can participate in InitRHI/ReleaseRHI)
  - InitRHI() creates all SHARED GPU buffers:
      PackedSplatBuffer + SRV
      SHBuffer + SRV
      ChunkBuffer + SRV
      IndexBuffer
      ClusterBuffer + SRV
      SplatClusterIndexBuffer + SRV
      LODSplatClusterIndexBuffer + SRV
  - After GPU upload, frees CPU-side cached arrays to save memory
  - ReleaseRHI() releases all shared buffers

What changes in FGaussianSplatGPUResources (Stage 2):
  - Remove all SHARED buffer members listed above
  - Add: TSharedPtr<FGaussianSplatRenderData> SharedData  (keeps ref alive)
  - Add getter methods that delegate to SharedData:
      GetPackedSplatBufferSRV() -> SharedData->PackedSplatBufferSRV
      GetSHBufferSRV()         -> SharedData->SHBufferSRV
      etc.
  - InitRHI() only creates PER-INSTANCE dynamic buffers
  - Keep: all sort buffers, visibility buffers, compaction buffers,
          indirect args, DummyWhiteTexture

What changes in rendering code (Stage 2):
  - GaussianSplatRenderer.cpp: Update shader parameter binding to use the
    getter methods instead of direct member access.
  - GaussianSplatting.cpp: Same - use getters for shared buffers.

Thread safety (Stage 2):
  - FGaussianSplatRenderData::InitRHI() is called on the render thread
  - Use FCriticalSection + bool bGPUInitialized to ensure only the first
    proxy triggers GPU upload; subsequent proxies wait until ready
  - The TSharedPtr ensures RenderData stays alive as long as any proxy
    references it (even if asset is GC'd during streaming)

Verification (Stage 2):
  In FGaussianSplatRenderData::InitRHI():
    UE_LOG(LogTemp, Log, TEXT("GaussianSplatRenderData: Created %d shared GPU buffers for asset '%s'"),
        BufferCount, *AssetName);

  In FGaussianSplatGPUResources::InitRHI():
    UE_LOG(LogTemp, Log, TEXT("GaussianSplatGPUResources: Created %d per-instance buffers (shared data: %s)"),
        BufferCount, *SharedData->GetAssetName());

  Expected log when alt+drag duplicating 1 asset 3 times:
    GaussianSplatRenderData: Created shared CPU data for asset 'Llama_V2' (211837 splats)
    GaussianSplatRenderData: Created 7 shared GPU buffers for asset 'Llama_V2'
    GaussianSplatGPUResources: Created 20 per-instance buffers (shared data: Llama_V2)
    GaussianSplatRenderData: Reusing existing shared data for asset 'Llama_V2'
    GaussianSplatGPUResources: Created 20 per-instance buffers (shared data: Llama_V2)
    GaussianSplatRenderData: Reusing existing shared data for asset 'Llama_V2'
    GaussianSplatGPUResources: Created 20 per-instance buffers (shared data: Llama_V2)

  Key check: "Created shared GPU buffers" should appear only ONCE per asset.
  Alt+drag 20 copies of the same asset should now be noticeably faster because
  only the small per-instance buffers are allocated (no splat packing, no bulk
  data upload).


--------------------------------------------------------------------------------
STAGE 3: Clean up and verify performance
--------------------------------------------------------------------------------

Purpose:
  Remove all verification logs added in Stage 1/2 (downgrade to Verbose).
  Do a final performance validation.

What to do:
  - Downgrade all Stage 1/2 UE_LOG lines to Verbose
  - Remove any leftover dead code in FGaussianSplatGPUResources
  - Ensure RenderData is properly invalidated when:
      * Asset is reimported
      * Nanite cluster hierarchy is rebuilt/cleared
      * Asset properties change in editor
  - Ensure RenderData is properly released when:
      * Last proxy using it is destroyed AND asset is unloaded
      * Editor closes / PIE ends

Verification (Stage 3):
  Manual testing checklist:
    [ ] Drag-drop single asset into level - renders correctly
    [ ] Alt+drag duplicate 1 copy - renders correctly, no freeze
    [ ] Alt+drag duplicate 20 copies - renders correctly, no freeze
    [ ] Delete all copies, undo, redo - no crash
    [ ] Two different assets in level - each has own shared data
    [ ] Reimport asset - shared data is recreated
    [ ] Toggle Nanite on/off on asset - shared data is recreated
    [ ] PIE play/stop - no crash or leak
    [ ] Save and reload level - renders correctly
