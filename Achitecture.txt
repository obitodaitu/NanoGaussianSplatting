[Screen Space Projected Error]
	*這一段有很大的機率有問題
	- Error = How much the LOD version is "wrong" compared to the original
	- LOD splat是相較於leaf cluster高階的點，原理是將leaf cluster的點merge成25%
	- merge的原理是平均position, rotation, color...2.3等資料
	- world space error: 原始點跟merge點相差的距離(看有多error)
	- screen space error: world space error做screen space轉換。我們在乎的是screen space的差異有多大而不是world space
	- Close > large projected error > low LOD
	- Far > small projected error > high LOD
	- MaxError = The actual error value STORED in each cluster (computed during import)
	- ErrorThreshold = The acceptable limit SET by user (runtime setting)
	- screen space error是在runtime下計算的

[Cluster System]
	- Process: Original points > Morton Code > Create Leaf Cluster > Create Parent Cluster > Create LOD splats > save to UGaussianSplatAsset > runtime culling
	- Morton Code: make points adjacent in 3D spce also adjacent in memory(1D Array). Z line adjacent as the final result in 3D space (faster for memory cache). Random order to spatially coherent
	- Create Leaf Cluster: Ater encoded Morton Code and arrange points into 1D Array, for each 128 points sliced into a cluster. Create bounds for frustum culling, screen space error
	- cluster是meta data加在point data上面
	- Create LOD splats: merge 4 points into one points(blending position, rotation and color value)
	- Total Point Count will be larger Than Original PLY. Total Points = Original Splats + LOD Splats
	- splat在decrease的同時，範圍也會變大，照理說不會disappear才對
	- ClusterCulling.usf是主要的LOD選擇算法
	- 紀錄每個cluster最大的離散程度就好 > 這就是MaxError的來由

[General]
	- this plugin has 2 modules(library/plugin), defined in GaussianSplatting.uplugin file
	- GaussianSplatting is Runtime Type(always), GaussianSplattingEditor is Editor type(editor only, won't ship with game)
	- header files use forward declarations when possible, .cpp files use #include
	- UPrimitiveComponent > can be place in the world and render
	- render workflow: game thread > scene proxy > render thread > shader > GPU
	- BufferRHIRef > GPU side都是這個type
	- GPU的memory跟CPU的是無法共用的
	- shader會用到Buffer裡面的資料比如ViewDataBuffer
	- Any data a shader need will need to create a buffer(in GPU memory)
	- GPU resource is just buffer(static buffer and dynamic buffer)
	- copy the data from RAM into VRAM
	- GPU variable has no type at all, untyped
	- CPU run C++ on RAM, GPU run HLSL on VRAM
	- CPU side UE will use the garbage collector to clean up the memory after actor got deleted
	- GPU side we will need to destroy by ourself
	- 如果沒有destroy會造成memory leak。CPU side是因為有garbage collector
	- UE本身就有一大串cmdlist跟GPU resource task，我只是把它reference過來，加入新funciton，再還回去
	- free VRAM when gaussian splat component got deleted
	- compute shader: not vertex shader, not pixel shader. pure computing, not related to output pixel
	- 用渲染的pixel去決定目前渲染的粗糙程度在畫面中能不能被接受 > 有多少pixel是錯的

--------------------------------------------------------------------------------------------
[Editor Module]
	- "PLYFileReader": Tools for process PLY binary files. Support reading and parsing data (position, rotation, scale, opacity, color)
	- "GaussianSplatAssetFactory": Import PLY file, build clusters and convert it into custom GaussianSplatAsset .uasset. Support reimport and mark file dirty
	- "GaussianSplatAssetTypeActions": Controls how the imported asset looks and behaves in the content browser editor

--------------------------------------------------------------------------------------------
[Runtime Module]
	[Data]
	- "GaussianDataTypes": Data stuct and enum for GaussianSplatAsset (FGaussianSplatData PLY raw data for each point)
	- "GaussianSplatAsset": .uasset which contains Gaussian Splatting data from PLY files. Fundamental UObject(pure data) + helper functions
	- we use custom .uasset "Gaussian Splat Asset"(serialized, UObject). Won't use original PLY file
	
	[Cluster]
	- "GaussianClusterTypes": define the point number of cluster in each level, helper functions (Encode Morton code...)
	- "GaussianClusterBuilder": define the cluster tree and point reduction ratio. Used in GaussianSplatAssetFactory
	
	[Actor & Component]
	- "GaussianSplatActor": Contain GaussianSplatComponent. Can be placed in the level
	- "GaussianSplatComponent": Game thread. Create component, expose user variables, include GaussianSplatAsset and GaussianSplatSceneProxy
	
	[GPU Data Management]
	- "GaussianSplatSceneProxy": Render thread. Mirror of the GaussianSplatComponent for GPU. Copy the data from bulk data on CPU(RAM), register GPU Resources(buffer), upload buffers to GPU(VRAM), shader use the buffer for calculation, destroy bufer after GaussianSplatComponent got removed
	- A thread safe, render-only copy of the component’s rendering data(UPrimitiveComponent). Render Thread won't touch UComponents
	- pure resource management
	- Buffer(pure GPU memory)
	- ShaderResourceViewRHIRef(SRV)(interpret the memory into type, shader read only)
	- UnorderedAccessViewRHIRef(UAV)(interpret the memory into type, shader read/write)
	- Static buffers: uploaded once from asset data, never change(PositionBuffer, OtherDataBuffer, SHBuffer, ClusterBuffer)
	- Dynamic buffers: allocated once but overwritten every frame by compute shaders(ViewDataBuffer, SortDistanceBuffer, SortKeysBuffer)
	- variable on GPU are untyped

	[Render Pipeline Integration]
	- "GaussianSplatViewExtension": register/unregister GaussianSceneProxy(we don't hook the render into pipeline here)
	- "GaussianSplatting": Module. Incude GaussianSplatViewExtension, build RDG, hook into render pipeline(currently before translucency)
	- each render pipeline stage usually contain lots of passes. RDG will be used for cleaning up the render
	
	[Render]
	- "GaussianSplatShaders": declare shader parameters struct, link with all shader files(.usf)
	- "GaussianSplatRenderer": connect parameters between GaussianSplatSceneProxy and GaussianSplatShaders. Add live view state(camera data) each frame

------------------------------------------------------------------------
[Shader]
	*Shaders will be fire through GaussianSplatRenderer's dispatch functions
	*Buffer were created through GaussianSceneProxy. Shader calculated based on the buffer(reference)
	- "CalcDistances.usf": For each splat, extracts Z coordinate from ViewDataBuffer. Converts depth to sortable uint
	- "RadixSort.usf": GPU gaussian splats sorting
	- "CalcViewData.usf": Transform 3D splat data to 2D screen-space data. Pass calculated result to ViewDataBuffer
	- "GaussianSplatRendering.usf": final render. Rasterrization, vertex shader, pixel shader
	- "GaussianSplatting.ush": some helper functions for loading data from buffer
	- "ClusterCulling.usf": Frustum culling and LOD selection at cluster level for Nanite-style optimization(screen space error)
	- "GaussianDataTypes.ush": some data type
	- "SphericalHarmonics.ush": spherical harmonics calculation
